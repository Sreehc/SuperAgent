# SuperAgent 前端 UI 优化需求说明

## 1. 背景

SuperAgent 当前已经具备较完整的企业级 AI 平台能力，包括登录与多租户权限、对话工作台、RAG 问答、知识库与文档管理、SSE 流式输出、Trace、Tools、治理、反馈、评测、成员、审计和运行时设置。

从现有代码看，项目已经从早期文档中的 Vue 方案演进为 React + Vite + React Router + Zustand + TanStack Query/Table + assistant-ui 的前端架构。聊天侧已经通过 `@assistant-ui/react` 的 `ExternalStoreRuntime` 接入现有 Spring Boot SSE 协议，因此不适合全量套用 assistant-ui 默认模板或 AI SDK 后端。

当前主要问题不是核心功能缺失，而是前端表达还偏工程后台原型：视觉层级相似、表格与表单交互粗糙、原生确认弹窗破坏体验、移动端布局不够理想，且部分后端已具备的 Agent/Trace/Tool/Eval/Document 能力没有在 UI 中充分呈现。

本需求文档聚焦“前端 UI 视觉与交互打磨 + 已有 API 能力补齐 + assistant-ui 集成正规化”。在不破坏现有功能的前提下，把 SuperAgent 打磨成更好看、更一致、更可用的 AI 工作台。

## 2. 目标

- 提升 SuperAgent 前端的整体视觉质量，使其从“可用后台”升级为“成熟的 AI 工作台与运行控制台”。
- 保留现有 React/Vite/Spring Boot SSE/Zustand 架构，避免因全量替换 assistant-ui 模板导致已有功能丢失。
- 正规化 assistant-ui 集成：保留现有外部运行时，把 RAG 引用、工具调用、Agent 步骤、Checkpoint、推荐追问等事件转化为更清晰的聊天 UI 部件。
- 统一全局设计系统，降低 `styles.css` 过大、页面级内联样式过多、表格/表单/弹窗体验不一致带来的维护成本。
- 改善高频页面的信息架构，包括聊天、知识库、文档详情、Trace、Tools、评测、设置和成员管理。
- 同步补齐 Tools、Eval、Governance、Document 等已有 API 的前端入口，让后端已具备的能力在 UI 中可发现、可操作。
- 提升可访问性、键盘可用性、移动端适配、加载/空/错误状态表达，并把移动端作为正式验收范围。

## 3. 用户

- 终端用户：通过对话工作台进行 RAG 问答、连续追问、查看引用来源和推荐问题。
- 知识库管理员：创建知识库、上传文档、查看解析状态、重处理文档、管理知识域和切块策略。
- 平台管理员：管理租户成员、角色、工具权限、模型配置、RAG 策略、Agent 设置和审计日志。
- 运营/质量人员：查看用户反馈、纠错建议、评测套件与运行结果，定位回答质量问题。
- 开发/运维人员：通过 Trace、Agent Run、Tool Call、模型调用和 Checkpoint 排查链路、延迟、失败和工具执行问题。

## 4. 使用场景

1. 终端用户进入 `/chat`，选择一个或多个知识库和执行模式，发送问题后看到 token-by-token 的流式回答，并能在右侧查看引用来源、执行链路和推荐追问。
2. 用户在同一会话中切换问题上下文，重命名、归档或删除会话，所有操作使用一致的弹窗、行内编辑或撤销反馈，而不是浏览器原生弹窗。
3. 知识库管理员在 `/knowledge/:knowledgeBaseId` 上传单个或多个文档，看到上传队列、解析状态、切块数量、失败原因和后续处理入口。
4. 管理员进入 `/documents/:documentId`，通过分栏或 tabs 查看解析文本、chunks、版本、图谱和任务日志。
5. 开发/运维人员从 `/traces` 进入某次 Trace 详情，查看阶段时间线、检索结果、模型调用、Agent Run、Tool Call 和 Checkpoint 关联信息。
6. 管理员在 `/tools` 查看工具能力、插件、密钥配置、最近工具调用和 Agent Runs，快速判断工具为什么不可用。
7. 质量人员在 `/evals` 创建或查看评测套件、运行记录、失败用例和报告摘要，用于回归测试和模型效果比较。
8. 管理员在 `/settings` 修改模型、RAG、重排序、Agent 和工具权限配置时，能看到风险说明、字段校验、保存状态和配置变更反馈。
9. 成员管理员在 `/members` 创建账号、发送邀请、修改角色、重置密码或移除成员时，能获得清晰的确认、校验和操作状态。
10. 移动端用户打开系统时，主聊天区域优先显示，会话栏和证据栏以抽屉方式打开，避免三栏内容在小屏上直接堆叠。
11. 质量人员从反馈纠错记录中一键生成评测用例，并将其纳入后续评测套件或回归运行。

## 5. 功能范围

### 5.1 全局 UI 与设计系统

- 拆分并整理全局样式，形成主题、布局、组件、聊天、Markdown、页面样式等清晰边界。
- 统一颜色、字体、间距、圆角、阴影、状态色和数值字体规则。
- 减少页面内联样式，把重复样式沉淀为共享 class 或组件变体。
- 建立通用页面骨架：标题区、操作区、筛选区、表格区、详情检查器、空状态和错误状态。
- 优化 loading 体验，更多使用匹配页面结构的 skeleton，而不是单一 spinner。
- 补充正式的空状态、错误状态、无权限状态和危险操作状态。

### 5.2 共享组件

- 优化 `DataTable`：排序标题使用真实按钮，行跳转使用链接或补齐键盘交互，支持筛选、分页、列宽、空态和长文本处理。
- 新增或完善 `ConfirmDialog`、`PromptDialog`、`FormField`、`SaveBar`、`DetailDrawer`、`StatusTimeline`、`MetricStrip` 等通用组件。
- 替换 `window.confirm`、`window.prompt` 和分散的危险操作逻辑。
- 统一按钮、图标按钮、tabs、badge、toast、dialog、drawer、select、input、textarea 的视觉和交互状态。
- 所有 icon-only 按钮必须具备可读的 `aria-label` 或 tooltip。

### 5.3 聊天工作台与 assistant-ui 集成

- 保留现有 `ExternalStoreRuntime` 和 Spring Boot SSE 协议，不重建为 assistant-ui 默认后端。
- 优化 assistant-ui Thread、Message、Composer 的视觉表现。
- 将 SSE 中的 `reference`、`recommendation`、`agent_step`、`tool_start`、`tool_result`、`checkpoint`、`resume`、`trace_stage` 映射为可读 UI。
- 消息区支持更清晰的用户/助手气泡、Markdown、代码块、表格、复制、反馈和错误状态。
- 工具调用、Checkpoint 和 Trace 阶段采用混合展示方案：聊天消息流中展示低噪音摘要，右侧检查器展示完整细节、输入输出、耗时、错误和关联链接。
- Composer 支持多知识库选择、执行模式选择、工具能力提示、发送、停止和输入状态反馈。
- 证据/Trace 右侧栏改为分组或 tabs，区分引用来源、执行链路、工具调用、Checkpoint 和推荐追问。
- 移动端将会话栏与证据栏改为 drawer/sheet，保证主线程优先。

### 5.4 知识库与文档页面

- 知识库列表增加更清晰的状态、文档数、更新时间、创建入口和空状态。
- 知识库详情优化上传体验，支持批量上传入口、上传队列、解析状态、失败原因和文档筛选。
- 文档详情补齐已有 API 入口，增加 tabs：解析文本、Chunks、版本、图谱、任务日志。
- 文档列表与文档详情中的时间、状态、分块数、版本号和任务日志使用统一格式。
- 删除、重处理等危险或高成本操作必须使用统一确认流程。

### 5.5 Trace、Tools 与 Agent 观测

- Trace 列表增加筛选、刷新、状态、耗时、模式、时间范围等操作。
- Trace 详情升级为时间线 + 详情检查器 + 关联检索 + 模型调用 + Agent Run/Tool Call 链接。
- Tools 页面从单一能力清单升级为多视图：工具能力、插件、密钥、工具调用、Agent Runs。
- 工具不可用时需要展示明确原因，例如未启用、缺少密钥、权限不足、域名受限或配置缺失。
- Tools 页面允许展示密钥“已设置/未设置”状态，但不展示密钥明文。
- Agent 步骤、Checkpoint、恢复状态和工具执行结果应在聊天和后台观测页中保持一致表达。

### 5.6 评测、反馈、治理、成员、审计与设置

- 评测页面支持查看套件、用例、运行记录、失败详情和报告摘要；补齐已有 API 的创建/运行入口。
- 反馈页面支持按评价类型筛选，增加详情抽屉、会话/消息/Trace 跳转入口，并支持从反馈纠错一键生成评测用例。
- 治理页面支持知识域与切块策略的创建、编辑、禁用和默认策略表达。
- 成员页面拆分创建账号、邀请成员、成员列表、待处理邀请，避免多个表单堆叠造成拥挤。
- 审计日志增加筛选、资源跳转、详情查看和导出预留入口。
- 设置页面使用规范 tabs，支持 URL 同步、字段校验、风险说明、保存状态、dirty state 和统一确认。
- 设置页高风险配置保存需要二次确认、审批或审计增强；审批规则可按配置风险等级分层实现。

### 5.7 可访问性、响应式与质量

- 所有可交互元素具备可见 focus 状态。
- 表格行、tabs、弹窗、抽屉、命令面板支持键盘操作。
- 表单控件具备 label、name、autocomplete、合适的 type/inputmode 和 inline error。
- 日期和数字格式统一封装，避免页面中分散 `new Date(...).toLocaleString(...)`。
- 移动端作为正式验收范围：优先保证聊天可用，管理后台允许降级为横向滚动表格或抽屉详情。
- 建立截图回归或设计基线，覆盖核心页面的桌面端和移动端状态，避免后续页面风格再次分散。
- 保留现有关键 `data-testid`，避免破坏 Playwright 测试。

## 6. 非目标

- 不重写后端、不改变现有 `/api/v1/*` API 契约、不替换 Spring Boot SSE 协议。
- 不全量套用 assistant-ui `create` 或 `init` 生成的默认项目结构。
- 不迁移到 Next.js App Router、AI SDK 默认后端、Assistant Cloud 或新的运行时平台。
- 不重新设计 Agent 编排、RAG 检索算法、模型服务、插件系统或数据库结构。
- 不做营销落地页、品牌官网、视觉 hero 或装饰性大改。
- 不引入大型后台组件库替换现有设计系统；允许按需引入轻量 UI 辅助依赖，例如虚拟列表、命令面板增强、图表或 JSON 编辑器。
- 不为了视觉优化删除或弱化现有 Trace、工具、评测、治理、审计等管理能力。
- 不在本阶段建设原生移动端应用。

## 7. 约束

- 前端技术栈保持 React + Vite + React Router + Zustand + TanStack Query/Table + assistant-ui。
- 聊天运行时必须继续兼容现有 `ExternalStoreRuntime`、Zustand store 和后端 SSE 事件。
- 路由路径和角色矩阵必须保持兼容，包括 `/chat`、`/chat/:sessionId`、`/knowledge`、`/documents/:documentId`、`/traces`、`/settings`、`/tools`、`/governance`、`/feedback`、`/evals`、`/members`、`/audit-logs` 等。
- 现有认证、多租户、`OWNER / ADMIN / MEMBER` 权限边界不能被前端改造破坏。
- 关键 `data-testid` 必须保留，避免现有 E2E 测试失效。
- 设计风格应符合控制台型产品：克制、密集、可扫描、面向高频工作流；避免营销式页面和过度装饰。
- 改造应分阶段推进，优先低风险共享组件和聊天工作台；发布方式为完成一组核心页面后统一发布。
- 任何新增依赖必须先确认是否与现有 Tailwind v4、Radix、assistant-ui、React 19 兼容；新增依赖应限定在虚拟列表、命令面板增强、图表、JSON 编辑器等明确 UI 辅助场景。
- 对用户生成内容和模型输出必须继续保持安全渲染，不能降低 Markdown 清洗和代码高亮的安全边界。

## 8. 成功标准

- 用户进入系统后，能在 `/chat` 直接完成新建会话、选择多个知识库、发送消息、停止生成、查看引用和继续追问。
- 回答仍然按 token 流式输出，且引用、工具调用、Agent 步骤、Checkpoint 和错误状态按混合方案可见、可理解。
- 浏览器原生 confirm/prompt 在主要业务流程中被统一弹窗或行内编辑替代。
- 管理后台主要页面拥有一致的页面骨架、表格、筛选、空态、错误态、加载态和危险操作反馈。
- 移动端聊天首屏优先显示消息线程，会话和证据通过抽屉访问。
- 关键页面的键盘导航和 focus 状态可用；主要表单具备 label、错误提示和保存状态。
- 核心页面有截图回归或设计基线覆盖，至少包含聊天、知识库、文档详情、Trace、Tools、评测、设置和成员页面。
- 前端构建、类型检查和现有单元测试/E2E 测试不因 UI 改造回退。
- 用户主观感知上，产品从“工程后台原型”提升为“成熟、清晰、可信的 AI 运行控制台”。
- 管理员能在 UI 中发现并使用更多已有后端能力，例如工具调用、插件、Agent Runs、文档 chunks、版本、图谱、评测运行等。

## 9. 已确认决策

1. 本轮 UI 优化包含视觉和交互打磨，并同步补齐 Tools、Eval、Governance、Document 等已有 API 的前端入口。
2. assistant-ui 聊天消息采用混合方案：聊天流展示执行摘要，右侧检查器展示工具调用、Checkpoint、Trace 阶段和排障细节。
3. 聊天工作台需要支持多知识库同时选择。
4. 移动端作为正式验收范围。
5. 需要建立截图回归或设计基线，避免后续页面风格再次分散。
6. 允许引入新的 UI 辅助依赖，例如虚拟列表、命令面板增强、图表或 JSON 编辑器。
7. 设置页的高风险配置保存需要二次确认、审批或审计增强。
8. 评测套件和运行记录需要支持从反馈纠错一键生成用例。
9. Tools 页面中的密钥配置允许前端展示“已设置/未设置”状态，但不展示密钥明文。
10. 本轮改造采用“完成一组核心页面后统一发布”的发布方式。
